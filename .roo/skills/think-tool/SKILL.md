# Think Tool - 智能技能调度器与思考工具

## 核心理念

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  双重模式：思考 + 调度                                         │
│                                                                 │
│  1. 思考模式：在工具调用之间停下来思考，而不是匆忙执行         │
│  2. 调度模式：分析任务，匹配最佳技能，协调执行                 │
│                                                                 │
│  "慢思考" 系统激活 → 更准确的推理 → 更好的决策 → 智能调度      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 双重模式选择

### 模式1：思考模式（原有功能）
**触发场景**：
- 任务涉及多个工具调用
- 需要权衡多个选项
- 决策有长期影响
- 涉及复杂逻辑推理
- 需要遵循特定规则或约束
- 用户请求模糊，需要澄清

### 模式2：调度模式（新增功能）
**触发场景**：
- 用户输入以 `/think-tool` 开头 + 任务描述
- 用户明确请求技能调度（例如“帮我搜索一下...”）
- 任务描述匹配某个技能的关键词
- 需要协调多个技能完成复杂任务

**模式选择流程**：
```
用户请求
    ↓
判断是否包含明确技能关键词或调度请求
    ↓
是 → 进入调度模式（技能匹配 → 调度执行）
否 → 进入思考模式（结构化思考 → 决策执行）
```

## 技能调度器功能

### 调度器架构

```
用户输入 "/think-tool + 任务描述"
    ↓
步骤1：任务解析
    - 提取任务描述中的关键词
    - 识别任务类型（搜索类、文档类、开发类、管理类等）
    - 判断是否需要联网
    ↓
步骤2：技能发现
    - 查看系统提示中的 `<available_skills>` 列表
    - 或使用 `list_files` 列出 `.roo/skills` 和 `.roo/skills-*` 目录
    - 获取每个技能的名称和描述
    ↓
步骤3：技能匹配
    - 根据关键词和任务类型匹配技能描述
    - 计算匹配度分数（基于关键词重合度和语义相关性）
    - 选择最佳匹配技能（或多个技能）
    ↓
步骤4：技能调度
    - 如果匹配到单个技能：直接调用该技能
    - 如果匹配到多个技能：提供技能选择建议或按顺序调用
    - 如果未匹配到技能：提供技能推荐或使用通用思考模式
    ↓
步骤5：结果返回
    - 整合技能执行结果
    - 提供执行总结
```

### 技能发现机制

调度器不依赖静态注册表，而是动态发现可用技能：

**方法1：使用系统提示中的技能列表**
- 每次对话开始时，系统提示中会包含 `<available_skills>` 部分
- 该列表提供了所有可用技能的 name 和 description
- 直接引用此列表进行匹配

**方法2：动态列出技能目录**
- 使用 `list_files` 工具列出 `.roo/skills` 和 `.roo/skills-*` 目录
- 读取每个技能的 SKILL.md 文件获取描述
- 适用于技能列表发生变化的情况

**方法3：用户提供技能列表**
- 用户可以显式提供需要调度的技能列表
- 适用于特定项目环境

### 技能匹配算法

1. **关键词提取**：从任务描述中提取中文和英文关键词
2. **描述匹配**：计算任务描述与技能描述的重合度
3. **分类过滤**：根据任务类型过滤技能分类
4. **网络需求**：检查网络可用性要求

**匹配度计算公式**（简化）：
```
匹配度 = (关键词匹配数 / 总关键词数) * 0.6 + (语义相似度) * 0.4
```

**语义相似度判断**：
- 如果任务描述包含技能名称，相似度 +0.3
- 如果任务描述包含技能关键词，相似度 +0.2 每个
- 如果技能描述明确提到任务类型，相似度 +0.2

### 技能调用机制

由于技能不能直接调用其他技能，think-tool 通过以下方式指导 AI 调用：

**方式1：直接使用技能功能**
- 对于简单任务，直接使用匹配技能的知识库和工具
- 示例：匹配到 web-search 技能，直接执行搜索

**方式2：使用 `new_task` 工具委派**
- 对于复杂任务，使用 `new_task` 工具委派给相应模式
- 示例：匹配到 task-master 技能，委派给 orchestrator 模式

**方式3：多技能协作**
- 对于需要多个技能的任务，按顺序调用或并行执行
- 示例：先使用 web-search 搜索，再使用 web-reader 阅读

## 思考工具功能（保留原有）

### 为什么需要思考工具

```
传统模式：
用户请求 ──▶ 立即行动 ──▶ 可能出错 ──▶ 修正 ──▶ 浪费时间和资源

思考模式：
用户请求 ──▶ 停下思考 ──▶ 分析方案 ──▶ 执行正确操作 ──▶ 高效完成
```

### 研究发现

Anthropic 的研究表明，在复杂任务中添加"思考"环节可以：

- **减少幻觉**：AI 更有可能发现推理中的错误
- **提高准确率**：特别是在多步骤任务中
- **改善工具使用**：减少不必要的工具调用
- **增强可解释性**：思考过程是可审查的

### 使用场景

**✅ 使用思考工具**：

- 任务涉及多个工具调用
- 需要权衡多个选项
- 决策有长期影响
- 涉及复杂逻辑推理
- 需要遵循特定规则或约束
- 用户请求模糊，需要澄清

**❌ 不使用思考工具**：

- 简单直接的查询
- 单一工具调用
- 任务步骤明确无需推理
- 快速原型或实验

### 思考工具协议

#### 触发条件

当遇到以下情况时，自动触发思考：

```markdown
1. 需要调用 3 个以上工具
2. 涉及文件修改或系统变更
3. 有多个可行方案需要选择
4. 需要考虑边界情况
5. 错误代价较高
```

#### 思考格式

```markdown
## 思考

### 当前状态
[描述当前任务进展和理解]

### 需要做什么
[明确下一步目标]

### 可选方案
1. 方案A：[描述] - 优点 / 缺点
2. 方案B：[描述] - 优点 / 缺点
3. 方案C：[描述] - 优点 / 缺点

### 决策依据
[为什么选择某个方案]

### 预期风险
[可能遇到的问题和应对]

### 执行计划
1. [步骤1]
2. [步骤2]
3. [步骤3]
```

## 示例

### 思考模式示例

#### 示例1：代码重构决策

```markdown
## 思考

### 当前状态
用户要求重构认证模块，当前代码散布在多个文件中。

### 需要做什么
将认证逻辑集中化，提高可维护性。

### 可选方案
1. 创建 AuthModule 类 - 优点：清晰的结构 / 缺点：需要大量修改
2. 创建工具函数集 - 优点：改动小 / 缺点：仍有分散风险
3. 使用中间件模式 - 优点：符合框架最佳实践 / 缺点：学习曲线

### 决策依据
选择方案3（中间件模式），理由：
- 符合 Express/Fastify 最佳实践
- 便于测试
- 扩展性好

### 预期风险
- 现有调用需要修改
- 需要更新测试用例

### 执行计划
1. 创建 authMiddleware.ts
2. 迁移核心逻辑
3. 更新路由配置
4. 更新测试
```

#### 示例2：多工具协调

```markdown
## 思考

### 当前状态
需要分析一个 GitHub 仓库的结构和代码质量。

### 需要做什么

### 可选方案
1. 顺序执行：克隆 → 分析 → 报告
2. 并行执行：同时获取多个维度的数据

### 决策依据
选择方案2，可以更快完成任务。

### 执行计划
1. 克隆仓库
2. 并行执行：
   - 目录结构分析
   - 代码统计
   - 依赖检查
   - 文档检查
3. 汇总报告
```

## 注意事项

### 通用注意事项
1. **不要过度思考** - 简单任务直接执行
2. **思考要具体** - 避免空洞的分析
3. **决策要明确** - 不要模棱两可
4. **保持简洁** - 思考过程不宜过长
5. **可追溯** - 重要决策要有记录
6. **及时记录** - 思考结果写入决策文件
7. **定期回顾** - 检验决策是否正确

### 调度器专用注意事项
1. **动态发现** - 技能列表可能变化，需定期更新
2. **匹配度阈值** - 设置合理匹配度阈值（建议 0.5）
3. **网络检查** - 调度前检查网络需求
4. **回退机制** - 确保有回退到思考模式的路径
5. **性能考虑** - 避免过长的匹配计算

### 兼容性注意事项
1. **向后兼容** - 原有思考功能完全保留
2. **模式自动切换** - 根据输入自动选择模式
3. **用户显式控制** - 支持用户指定模式
4. **技能版本兼容** - 注意不同技能版本的兼容性

## 故障排除

### 常见问题

1. **调度器无法匹配技能**
   - 检查技能发现机制是否正常工作
   - 验证关键词提取是否准确
   - 尝试降低匹配度阈值

2. **思考模式触发过于频繁**
   - 调整触发条件
   - 明确简单任务直接执行

3. **技能调用失败**
   - 检查技能是否可用
   - 验证网络连接（如果需要）
   - 使用回退方案

4. **性能问题**
   - 优化匹配算法
   - 缓存技能列表
   - 减少不必要的计算
