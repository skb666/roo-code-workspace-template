# 上下文工程策略

基于 Anthropic "Effective Context Engineering for AI Agents" 的核心方法。

## 核心理念

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   上下文 = 有限资源 + 注意力预算                                │
│                                                                 │
│   目标：找到最小的高信号 Token 集合                             │
│         最大化期望结果的可能性                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 为什么上下文工程很重要

### 上下文衰减（Context Rot）

```
Token 数量 ──────────────────────────────▶

质量 ████████████▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░
     │
     └─ 随着上下文增长，模型准确召回信息的能力下降

原因：
1. Transformer 架构：n² 成对关系，注意力被稀释
2. 训练分布：模型在短序列上训练更多
3. 位置编码：扩展到更长序列有精度损失
```

### 与 Prompt Engineering 的区别

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|---------------------|
| 焦点 | 如何写好提示 | 如何管理整体上下文 |
| 范围 | 系统提示 | 提示 + 工具 + 历史 + 外部数据 |
| 性质 | 离散任务 | 迭代过程 |
| 阶段 | 每次推理前 | 每次推理决策时 |

## 上下文组件优化

### 1. 系统提示（System Prompt）

**原则**：

```
┌─────────────────────────────────────────────────────────────────┐
│  找到"最佳高度"（Goldilocks Zone）                              │
│                                                                 │
│  太低（模糊）              太高（脆弱）                         │
│       │                        │                                │
│       ▼                        ▼                                │
│  "做这个任务"            "如果 X 则 Y，                          │
│  没有具体指导              否则如果 A 则 B..."                   │
│                                                                 │
│                    最佳高度                                     │
│                       │                                         │
│                       ▼                                         │
│              足够具体以指导行为                                  │
│              足够灵活以适应变化                                  │
└─────────────────────────────────────────────────────────────────┘
```

**结构建议**：

```markdown
<background_information>
  项目背景、约束条件
</background_information>

<instructions>
  核心指令，使用清晰、直接的语言
</instructions>

## Tool guidance
  工具使用指南

## Output description
  输出格式要求
```

**最佳实践**：

- 从最小提示开始测试
- 根据失败模式添加指令
- 不要堆砌边缘情况

---

### 2. 工具（Tools）

**工具设计原则**：

```markdown
1. 自包含
   - 每个工具有明确的单一职责
   - 功能不重叠

2. 健壮
   - 清晰的错误处理
   - 明确的使用说明

3. 清晰
   - 参数名直观
   - 包含使用示例
```

**常见失败模式**：

```
❌ 工具集臃肿
   → 功能重叠
   → Agent 难以选择
   → 如果人类都无法确定用哪个，Agent 更无法判断

✅ 精简的最小可行工具集
   → 每个工具有明确边界
   → 易于选择和维护
```

---

### 3. 示例（Examples）

**原则**：

```markdown
❌ 不要：堆砌边缘情况试图覆盖所有规则

✅ 要：精选多样化的典型示例

原因：
- 示例是 LLM 的"图片"，胜过千言万语
- 典型示例比边缘情况规则更有效
```

## 动态上下文策略

### 1. Just-in-Time Retrieval（按需检索）

**概念**：不在预先加载所有信息，而是维护轻量引用，按需加载。

```
┌─────────────────────────────────────────────────────────────────┐
│                    传统方式 vs JIT 方式                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统：                                                         │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 预先加载所有相关文件 → 可能相关 → 可能相关 → ...    │       │
│  │ 上下文消耗大，很多可能不需要                        │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  JIT：                                                          │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                   │
│  │ 文件路径 │────▶│ 按需读取 │────▶│ 处理内容 │                   │
│  └─────────┘     └─────────┘     └─────────┘                   │
│  只维护引用，按需加载                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**实施方法**：

```markdown
1. 维护轻量引用
   - 文件路径
   - 存储的查询
   - Web 链接
   - 数据库 ID

2. 使用工具动态加载
   - glob 定位文件
   - grep 搜索内容
   - head/tail 预览大文件
   - SQL 查询数据库

3. 利用元数据
   - 文件大小暗示复杂度
   - 命名约定暗示用途
   - 时间戳暗示相关性
```

**示例**：

```javascript
// ❌ 传统方式：预先加载所有
const allFiles = loadAllFiles(projectPath);
const context = allFiles.map(f => f.content).join('\n');
// 上下文可能达到数十万 Token

// ✅ JIT 方式：按需加载
const relevantPaths = glob('**/auth*.ts');
const fileToRead = relevantPaths[0];
const content = readFile(fileToRead);
// 只消耗必要的 Token
```

### 2. Progressive Disclosure（渐进式披露）

**概念**：让 Agent 通过探索逐步发现相关信息。

```
探索路径示例：

1. 查看项目根目录
   → 发现 src/ 和 tests/ 文件夹
   → 推测 src/ 是源代码

2. 查看 src/ 目录
   → 发现 auth/ 模块
   → 推测与认证相关

3. 查看 auth/ 目录
   → 发现 login.ts, register.ts
   → 理解认证模块结构

每一步只保留必要信息在上下文中
```

**好处**：

- 避免信息过载
- 利用结构元数据
- 保持上下文聚焦

---

### 3. Hybrid Strategy（混合策略）

**概念**：结合预加载和按需检索。

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   预加载（速度）              按需检索（灵活性）                │
│        │                            │                          │
│        ▼                            ▼                          │
│   ┌─────────┐                 ┌─────────────┐                 │
│   │CLAUDE.md│                 │ glob + grep │                 │
│   │项目概览 │                 │ 按需探索    │                 │
│   └─────────┘                 └─────────────┘                 │
│                                                                 │
│   静态、核心信息              动态、特定信息                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Roo 中的应用**：

```markdown
预加载：
- 项目结构概览
- 核心配置文件
- CLAUDE.md（如果有）

按需检索：
- 具体代码文件
- 测试文件
- 文档片段
```

## 长时任务上下文策略

### 1. Compaction（压缩）

**概念**：当上下文接近限制时，总结历史，重新开始。

```
压缩前：
┌───────────────────────────────────────────────────────┐
│ 完整对话历史 (80k tokens)                              │
│ - 详细工具调用和结果                                   │
│ - 完整的代码讨论                                       │
│ - 所有中间步骤                                         │
└───────────────────────────────────────────────────────┘
                          │
                          ▼ 压缩
                          │
压缩后：
┌───────────────────────────────────────────────────────┐
│ 关键信息摘要 (5k tokens)                              │
│ - 架构决策                                            │
│ - 未解决的问题                                        │
│ - 实现细节                                            │
│ - 最近访问的 5 个文件                                 │
└───────────────────────────────────────────────────────┘
```

**压缩内容选择**：

```markdown
保留：
✅ 架构决策及其理由
✅ 未解决的 Bug
✅ 关键实现细节
✅ 重要约束和依赖

丢弃：
❌ 冗余的工具输出
❌ 中间调试信息
❌ 已解决且无需追溯的讨论
```

**最轻量的压缩：工具结果清理**

```markdown
深层历史中的工具调用结果可以：
1. 保留关键摘要
2. 丢弃完整输出

例如：
- "文件读取成功，发现 3 个函数定义"
- 而非完整的文件内容
```

### 2. Structured Note-taking（结构化笔记）

**概念**：Agent 定期将笔记持久化到上下文窗口之外。

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  上下文窗口                        外部存储                      │
│  ┌──────────────┐                ┌──────────────┐              │
│  │ 当前工作     │                │ NOTES.md     │              │
│  │ (活跃记忆)   │ ────写入────▶  │ 项目状态     │              │
│  │              │                │ 决策记录     │              │
│  │              │ ◀────读取────  │ 进度跟踪     │              │
│  └──────────────┘                └──────────────┘              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**应用场景**：

```markdown
1. TODO 列表
   - 跟踪复杂任务进度
   - 跨会话维护

2. 项目 NOTES.md
   - 架构决策
   - 已知问题
   - 待办事项

3. 状态文件
   - 功能完成状态
   - 测试覆盖情况
```

**示例：Claude 玩 Pokémon**

```markdown
Agent 自动维护的笔记：

## 训练进度
- 皮卡丘等级：15/20
- 当前位置：Route 1
- 训练步数：1234

## 已解锁成就
- 获得初始宝可梦
- 战胜道馆馆主 1

## 战斗策略笔记
- 对岩石系用电击效果差
- 对水系用电击效果好
```

### 3. Sub-agent Architecture（子代理架构）

**概念**：主代理协调，子代理执行深度工作，返回精简摘要。

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   ┌───────────────────┐                                        │
│   │     主 Agent       │ ← 保持轻量，专注协调                   │
│   │   (上下文干净)     │                                        │
│   └─────────┬─────────┘                                        │
│             │                                                   │
│     ┌───────┼───────┬───────────┐                              │
│     │       │       │           │                              │
│     ▼       ▼       ▼           ▼                              │
│  ┌─────┐ ┌─────┐ ┌─────┐   ┌─────┐                            │
│  │子A  │ │子B  │ │子C  │   │子D  │  ← 可探索大量上下文        │
│  │搜索 │ │分析 │ │实现 │   │测试 │                            │
│  └──┬──┘ └──┬──┘ └──┬──┘   └──┬──┘                            │
│     │       │       │          │                               │
│     ▼       ▼       ▼          ▼                               │
│   摘要    摘要    摘要       摘要    ← 只返回 1-2k Token       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**在 Roo 中的实现**：

```javascript
// 主 Agent（协调器）
new_task({
  mode: "code",
  message: "搜索项目中所有认证相关的代码，返回文件列表和关键发现摘要"
})
// 子 Agent 使用大量上下文探索
// 返回精简摘要

// 主 Agent 收到摘要，继续决策
// 不需要看到子 Agent 的完整探索过程
```

**好处**：

```markdown
1. 上下文隔离
   - 子 Agent 可以使用大量 Token 探索
   - 主 Agent 只接收精简摘要

2. 清晰的职责分离
   - 主 Agent：决策和协调
   - 子 Agent：深度技术工作

3. 可并行执行
   - 多个子 Agent 同时工作
   - 各自独立探索
```

## 策略选择指南

| 任务特征 | 推荐策略 |
|----------|----------|
| 需要多次来回对话 | Compaction |
| 有明确里程碑的迭代开发 | Structured Note-taking |
| 复杂研究和分析 | Sub-agent Architecture |
| 动态数据探索 | Just-in-Time Retrieval |

---

## 上下文工程检查清单

```markdown
每次推理前检查：

1. 系统提示
   - [ ] 清晰直接？
   - [ ] 不会太具体（脆弱）？
   - [ ] 不会太模糊？

2. 工具集
   - [ ] 精简？
   - [ ] 无功能重叠？
   - [ ] 参数清晰？

3. 示例
   - [ ] 典型且多样？
   - [ ] 不是边缘情况堆砌？

4. 动态内容
   - [ ] 是否可以按需加载？
   - [ ] 是否有轻量引用？

5. 历史管理
   - [ ] 是否需要压缩？
   - [ ] 是否需要笔记持久化？
   - [ ] 是否可以用子代理？
```
