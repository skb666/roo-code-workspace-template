# 增量执行方法

基于 Anthropic "Effective Harnesses for Long-Running Agents" 的核心洞察。

## 核心问题

长时运行 Agent 面临的核心挑战：

```
┌─────────────────────────────────────────────────────────────────┐
│  Agent 必须在离散会话中工作                                      │
│  每个新会话开始时，没有之前会话的记忆                             │
│  就像工程师轮班工作，每个新工程师来时没有上一班的记忆             │
└─────────────────────────────────────────────────────────────────┘
```

## 四大失败模式及解决方案

### 失败模式 1：过早宣布完成

**症状**：Agent 看到一些进度就认为任务完成

**根本原因**：缺乏明确的完成标准

**解决方案**：

```json
// 功能清单文件：feature-status.json
{
  "features": [
    {
      "id": "F001",
      "category": "functional",
      "description": "用户可以创建新聊天",
      "steps": [
        "点击新建聊天按钮",
        "验证创建了新会话",
        "验证聊天区域显示欢迎状态"
      ],
      "passes": false,
      "verified_by": null,
      "verified_at": null
    }
  ]
}
```

**关键规则**：

```markdown
⚠️ 禁止删除或编辑测试用例
⚠️ 只能修改 passes 字段（false → true）
⚠️ 只有在完整验证后才能标记为 true
```

---

### 失败模式 2：一次做太多

**症状**：尝试一次性完成所有功能，导致上下文耗尽

**根本原因**：缺乏增量执行约束

**解决方案**：

```markdown
## 增量执行原则

每次会话只做一件事：
1. 从功能清单中选择优先级最高的未完成功能
2. 实现该功能
3. 验证该功能
4. 更新状态文件
5. 提交 Git

❌ 禁止：同时在多个功能上工作
❌ 禁止：实现功能 A 时顺便修改功能 B
✅ 要求：每个功能独立、完整的开发周期
```

---

### 失败模式 3：留下混乱状态

**症状**：会话结束时留下半成品、未记录的变更

**根本原因**：缺乏状态清理和记录机制

**解决方案**：

```markdown
## 会话结束检查清单

每次会话结束前必须完成：
- [ ] 所有代码变更已提交
- [ ] 提交信息清晰描述变更内容
- [ ] 功能状态已更新
- [ ] 会话日志已更新
- [ ] 代码处于可运行状态
- [ ] 无遗留的调试代码
- [ ] 无临时的硬编码值

## Git 提交规范

格式：
<type>(<scope>): <subject>

<body>

类型：
- feat: 新功能
- fix: 修复
- refactor: 重构
- test: 测试
- docs: 文档
- chore: 构建/工具

示例：
feat(auth): 添加 JWT Token 生成

- 实现 token 生成逻辑
- 添加过期时间配置
- 编写单元测试

Closes #123
```

---

### 失败模式 4：标记完成但未测试

**症状**：写完代码就标记完成，没有端到端验证

**根本原因**：缺乏强制测试机制

**解决方案**：

```markdown
## 测试金字塔

        ┌─────────────┐
        │  E2E 测试   │  ← 每个功能必须通过
        └─────────────┘
      ┌───────────────────┐
      │   集成测试        │  ← 模块间交互
      └───────────────────┘
    ┌─────────────────────────┐
    │      单元测试           │  ← 函数/方法级别
    └─────────────────────────┘

## 强制测试流程

1. 编写功能代码
2. 编写/运行单元测试
3. 运行集成测试
4. 运行端到端测试（如适用）
5. 只有所有测试通过才能标记完成

## 测试工具

对于 Web 应用：
- 使用 Puppeteer/Playwright 进行自动化测试
- 截图验证 UI 状态
- 模拟用户操作流程
```

---

## 会话启动流程

每次新会话开始时，执行以下步骤快速恢复状态：

```markdown
## 会话启动检查清单

### 1. 确认工作目录
```bash
pwd
```

### 2. 阅读项目状态
```
- 读取 .roo/project-state/context.md
- 读取 .roo/project-state/session-log.md
- 读取 .roo/project-state/feature-status.json
```

### 3. 检查 Git 历史
```bash
git log --oneline -10
git status
```

### 4. 验证项目状态
```
- 如果有 init.sh，运行它
- 启动开发服务器
- 验证基本功能正常
```

### 5. 选择下一个任务
```
- 从功能清单选择优先级最高的未完成功能
- 确认该功能的验证步骤
- 开始执行
```
```

## 最佳实践总结

```
┌─────────────────────────────────────────────────────────────────┐
│                      增量执行黄金法则                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 一次一个功能                                                │
│     - 选择一个功能 → 实现 → 验证 → 提交                         │
│                                                                 │
│  2. 每次留下干净状态                                            │
│     - 代码可运行                                                │
│     - 测试通过                                                  │
│     - 已提交                                                    │
│     - 已记录                                                    │
│                                                                 │
│  3. 验证是必须的                                                │
│     - 单元测试 + 端到端测试                                     │
│     - 所有测试通过才能标记完成                                  │
│                                                                 │
│  4. 状态可恢复                                                  │
│     - 功能清单 (feature-status.json)                            │
│     - 会话日志 (session-log.md)                                 │
│     - 决策记录 (decisions.md)                                   │
│     - Git 历史                                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
