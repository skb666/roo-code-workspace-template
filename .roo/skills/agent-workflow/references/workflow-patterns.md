# 六种工作流模式详解

基于 Anthropic "Building Effective Agents" 的核心模式。

## 模式概览

```
复杂度递增 →
┌──────────────┬──────────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│Prompt Chaining│   Routing    │Parallelization│Orchestrator  │Evaluator     │Autonomous    │
│              │              │              │  -Workers    │-Optimizer    │   Agent      │
└──────────────┴──────────────┴──────────────┴──────────────┴──────────────┴──────────────┘
     简单                                                                              复杂
```

---

## 1. Prompt Chaining（提示链）

### 概念

将任务分解为顺序步骤，每个 LLM 调用处理前一步的输出。

### 架构图

```
输入 ──▶ LLM 1 ──▶ 检查点 ──▶ LLM 2 ──▶ 检查点 ──▶ LLM 3 ──▶ 输出
```

### 适用场景

- 任务可清晰分解为固定子任务
- 每个子任务可以更简单的提示完成
- 愿意用延迟换取更高准确性

### 示例

```javascript
// 任务：生成营销文案并翻译
步骤1: 生成中文营销文案
步骤2: 将文案翻译成英文
步骤3: 校对英文文案

// 在 Roo 中的实现
new_task({ mode: "code", message: "步骤1..." })
  → 完成后
new_task({ mode: "code", message: "基于步骤1结果，执行步骤2..." })
  → 完成后
new_task({ mode: "code", message: "基于步骤2结果，执行步骤3..." })
```

### 检查点设计

```python
# 在每个步骤之间加入检查
if not step1_result.meets_criteria:
    # 调整或重试
    retry_or_adjust()
else:
    proceed_to_step2()
```

## 2. Routing（路由）

### 概念

对输入分类，导向专门的下游任务。

### 架构图

```
                    ┌──▶ 专门处理 A ──▶ 输出 A
                    │
输入 ──▶ 分类器 ────┼──▶ 专门处理 B ──▶ 输出 B
                    │
                    └──▶ 专门处理 C ──▶ 输出 C
```

### 适用场景

- 复杂任务有不同类别需要分别处理
- 分类可以准确完成（LLM 或传统分类器）
- 优化某类输入不应影响其他类

### 示例

```javascript
// 任务：处理客服查询
路由规则：
- 一般问题 → Ask 模式（使用知识库回答）
- 退款请求 → Code 模式（调用退款 API）
- 技术支持 → Debug 模式（诊断问题）

// 在 Roo 中的实现
// 1. 先分析输入类型
// 2. switch_mode 切换到合适模式
// 3. 由专门模式处理
```

### 路由策略

| 策略 | 说明 | 适用 |
|------|------|------|
| LLM 路由 | 用 LLM 判断类型 | 类别复杂、模糊边界 |
| 规则路由 | 用关键词/正则匹配 | 类别明确、规则清晰 |
| 模型路由 | 用小模型分类 | 类别固定、追求效率 |

## 3. Parallelization（并行化）

### 概念

多个 LLM 同时工作，输出聚合。

### 架构图

```
         ┌──▶ LLM A ──┐
         │            │
输入 ────┼──▶ LLM B ───┼──▶ 聚合 ──▶ 输出
         │            │
         └──▶ LLM C ──┘
```

### 两种变体

#### 变体 1：Sectioning（分段）

将任务分解为独立子任务并行执行。

```
任务：开发用户认证系统

并行执行：
├── 前端登录页面开发
├── 后端 API 开发
└── 数据库设计

聚合：合并所有组件
```

#### 变体 2：Voting（投票）

同一任务执行多次，获取多样化结果。

```
任务：代码安全审查

并行执行：
├── 审查员 A：检查注入漏洞
├── 审查员 B：检查权限问题
└── 审查员 C：检查数据泄露

聚合：合并所有发现的问题
```

### 在 Roo 中的实现

```javascript
// 并行启动多个子任务
// Roo 支持在单次响应中并行调用 new_task

// 分段示例
[
  new_task({ mode: "code", message: "开发前端登录页面..." }),
  new_task({ mode: "code", message: "开发后端 API..." }),
  new_task({ mode: "code", message: "设计数据库..." })
]

// 等待所有完成，然后聚合结果
```

## 4. Orchestrator-Workers（协调器-工作器）

### 概念

中心 LLM 动态分解任务，分配给工作 LLM，综合结果。

### 架构图

```
                    ┌──▶ Worker A ──┐
                    │              │
输入 ──▶ 协调器 ────┼──▶ Worker B ───┼──▶ 协调器综合 ──▶ 输出
                    │              │
                    └──▶ Worker C ──┘
         ▲                              │
         └──────────────────────────────┘
              （可迭代多轮）
```

### 与并行化的关键区别

| 维度 | 并行化 | 协调器-工作器 |
|------|--------|---------------|
| 子任务 | 预先定义 | 动态确定 |
| 灵活性 | 低 | 高 |
| 复杂度 | 低 | 高 |

### 适用场景

- 无法预测需要的子任务
- 子任务取决于具体输入
- 需要多轮迭代

### 示例

```javascript
// 任务：解决 GitHub Issue
协调器分析：
- 需要修改哪些文件？（动态确定）
- 每个文件需要什么改动？
- 改动之间有依赖关系吗？

Worker 分配：
- Worker 1：修改 config.js
- Worker 2：修改 api.js
- Worker 3：更新测试文件

协调器综合：
- 检查改动是否一致
- 运行测试验证
- 可能需要新一轮修改
```

### 在 Roo 中的实现

```javascript
// 协调器模式的核心实现
// 1. 分析任务，确定需要的子任务
update_todo_list({
  todos: `
  - [-] 分析任务需求
  - [ ] 子任务1：xxx
  - [ ] 子任务2：xxx
  - [ ] 综合结果
  `
})

// 2. 委派给工作模式
new_task({ mode: "code", message: "子任务1详细指令..." })
new_task({ mode: "code", message: "子任务2详细指令..." })

// 3. 收集结果并综合
// 分析每个子任务结果，确定后续步骤
```

## 5. Evaluator-Optimizer（评估器-优化器）

### 概念

一个 LLM 生成响应，另一个提供评估反馈，循环迭代。

### 架构图

```
                    ┌──────────────────────┐
                    │                      │
                    ▼                      │
输入 ──▶ 生成器 ──▶ 输出 ──▶ 评估器 ──▶ 反馈 ─┘
                      │
                      ▼
                   最终输出
                  （当评估通过）
```

### 适用场景

- 有清晰的评估标准
- 迭代改进有明确价值
- LLM 能理解和应用反馈

### 示例

```javascript
// 任务：文学翻译
生成器：翻译文本
评估器：评估翻译质量，指出改进点
  → "这段翻译丢失了原文的幽默感，建议..."
生成器：根据反馈改进
评估器：再次评估
  → "改进了，但这个成语翻译还可以优化..."
... 循环直到满意 ...

// 在 Roo 中的实现
循环：
  1. Code 模式实现功能
  2. Debug/Ask 模式评估
  3. 如果不满意，Code 模式根据反馈修改
  4. 重复直到通过
```

### 评估标准设计

```markdown
好的评估标准应该：
- 具体：明确指出好/坏的标准
- 可操作：反馈能转化为具体行动
- 一致：多次评估结果一致
```

## 6. Autonomous Agent（自主代理）

### 概念

LLM 在循环中自主使用工具，根据环境反馈行动。

### 架构图

```
┌─────────────────────────────────────────────┐
│                                             │
│    ┌──────────┐                            │
│    │          │                            │
│    │   LLM    │──▶ 工具调用 ──▶ 环境反馈   │
│    │          │                            │
│    └──────────┘                            │
│         ▲                                  │
│         │                                  │
│         └──────────────────────────────────┘
│                  循环
└─────────────────────────────────────────────┘
```

### 适用场景

- 开放式问题，难以预测步骤数
- 无法硬编码固定路径
- 可信任 LLM 的决策能力

### 示例

```javascript
// 任务：解决 SWE-bench Issue
Agent 循环：
1. 阅读 Issue 描述
2. 探索代码库找到相关文件
3. 理解当前实现
4. 规划修改方案
5. 实施修改
6. 运行测试
7. 如果测试失败，分析原因，返回步骤4
8. 如果测试通过，完成任务
```

### 在 Roo 中的实现

```javascript
// Roo 的 Orchestrator 模式天然支持这种模式
// 关键是设计好工具和指导原则

核心能力：
- new_task：委派子任务
- switch_mode：切换专业模式
- update_todo_list：跟踪进度
- attempt_completion：完成任务

关键设计：
1. 清晰的工具定义
2. 明确的停止条件
3. 沙盒环境测试
4. 适当的护栏
```

### 安全考虑

```
自主性越高：
- 能力越强
- 成本越高
- 错误累积风险越大

建议：
- 在沙盒环境充分测试
- 设置最大迭代次数
- 关键操作需要人工确认
- 记录完整执行日志
```

## 组合使用

这些模式可以组合：

```javascript
// 组合示例：复杂软件开发项目

// 1. Routing：确定项目类型
switch_mode({ mode: "architect" })
  → 确定是 Web 应用项目

// 2. Orchestrator-Workers：分解任务
  → 前端、后端、测试

// 3. Parallelization：并行开发
new_task({ mode: "code", message: "前端开发..." })
new_task({ mode: "code", message: "后端开发..." })

// 4. Evaluator-Optimizer：代码审查和优化
switch_mode({ mode: "debug" })
  → 发现问题
switch_mode({ mode: "code" })
  → 修复问题

// 5. Prompt Chaining：部署流程
  → 构建 → 测试 → 部署
```
