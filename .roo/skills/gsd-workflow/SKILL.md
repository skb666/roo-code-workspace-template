# GSD Workflow - 规范驱动开发框架

## 快速参考

|| 阶段 | 命令 | 核心输出 |
|------|------|----------|
| 初始化 | `/gsd:init` | PROJECT.md, REQUIREMENTS.md, ROADMAP.md |
| 讨论 | `/gsd:discuss N` | 阶段偏好、技术选型决策 |
| 规划 | `/gsd:plan N` | PLAN.md（原子任务计划） |
| 执行 | `/gsd:execute N` | 原子提交、功能实现 |
| 验证 | `/gsd:verify N` | 验证报告、状态更新 |
| 快速 | `/gsd:quick` | 简化流程处理小任务 |

## 核心解决的问题

### Context Rot（上下文腐烂）

```
传统 AI 编程：

会话开始 ────────────────────────────────────▶
质量      ████████████▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░
          │
          └─ 随着对话进行，上下文窗口被填满
             AI 的输出质量逐渐下降

GSD 解决方案：

每个任务独立上下文（200K tokens）
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 任务 A  │  │ 任务 B  │  │ 任务 C  │
│ 全新上下文│  │ 全新上下文│  │ 全新上下文│
└─────────┘  └─────────┘  └─────────┘
质量始终稳定
```

## 使用场景

**✅ 使用此技能**：

- 目标明确的项目开发
- 需要交付生产级代码
- 跨多个会话的复杂任务
- 需要 Git 历史清晰可追溯
- 团队协作项目

**❌ 不使用此技能**：

- 快速原型验证
- 探索性编码
- 简单的单次交互任务
- 不确定目标的学习项目

### PLAN.md - 原子任务计划

```markdown
# Phase 1 执行计划

## 任务依赖图
```
[A1: 数据库设计] ──┬──▶ [A3: API 开发]
                  │
[A2: 前端骨架] ───┴──▶ [A4: 集成测试]
```

## 原子任务

### A1: 数据库设计
- **描述**: 设计并创建数据库表结构
- **预计时间**: 2h
- **文件**: `src/db/schema.sql`
- **验收**: 表创建成功，索引正确
- **状态**: pending

### A2: 前端骨架
- **描述**: 创建前端项目基础结构
- **预计时间**: 1h
- **文件**: `src/frontend/`
- **验收**: 项目可运行
- **状态**: pending

### A3: API 开发
- **描述**: 开发认证 API
- **依赖**: A1
- **预计时间**: 4h
- **文件**: `src/api/auth.ts`
- **验收**: API 测试通过
- **状态**: pending
```

## 工作流程

### 阶段一：初始化

**目的**: 建立项目基础文档和状态管理

**步骤**:

1. 创建项目状态目录
```
.roo/gsd-state/
├── PROJECT.md
├── REQUIREMENTS.md
├── ROADMAP.md
├── PLAN.md
├── STATE.md
└── decisions/
    └── ADR-001.md
```

2. 收集需求信息
```markdown
通过提问明确：
- 项目目标是什么？
- 核心用户是谁？
- 主要功能有哪些？
- 技术约束是什么？
- 交付时间？
```

3. 生成核心文档
```
- PROJECT.md（项目愿景）
- REQUIREMENTS.md（需求清单，带 REQ-ID）
- ROADMAP.md（阶段划分）
```

---

### 阶段二：讨论（Discuss）

**目的**: 明确实现偏好和技术选型

**讨论内容**:

```markdown
## Phase N 讨论要点

### 技术选型
- 框架选择：React vs Vue vs Svelte
- 状态管理：Redux vs Zustand vs Jotai
- 样式方案：CSS Modules vs Tailwind vs styled-components

### 实现偏好
- 编码风格
- 文件组织
- 命名约定

### 视觉风格（如适用）
- 配色方案
- 组件库选择
- 响应式策略

### API 设计
- REST vs GraphQL
- 认证方案
- 错误处理
```

**输出**: `decisions/ADR-XXX.md`（架构决策记录）

---

### 阶段三：规划（Plan）

**目的**: 生成可执行的原子任务计划

**规划原则**:

```
1. 原子性
   - 每个任务可独立完成
   - 每个任务有明确的验收标准

2. 可并行
   - 识别任务依赖关系
   - 独立任务可并行执行

3. 可验证
   - 每个任务完成后可验证
   - 验证不依赖其他未完成任务
```

**依赖分析**:

```
波浪式并行执行：

第一波（无依赖）:
[A1] ──┐
[A2] ──┼──▶ 并行执行
[A3] ──┘

第二波（依赖第一波）:
[A4] ──┐
[A5] ──┼──▶ 等待依赖完成后并行
[A6] ──┘

第三波（依赖第二波）:
[A7] ──▶ 等待依赖完成后执行
```

---

### 阶段四：执行（Execute）

**目的**: 按计划执行原子任务

**执行原则**:

```markdown
1. 独立上下文
   - 每个任务使用独立的上下文窗口
   - 通过 STATE.md 恢复上下文

2. 原子提交
   - 每个任务完成后单独提交
   - 提交信息格式：
     feat(scope): 简短描述
     
     - 详细说明
     - 相关 REQ-ID

3. 验证优先
   - 完成后立即验证
   - 不依赖人工确认

4. 状态同步
   - 实时更新 STATE.md
   - 记录关键决策
```

**执行模板**:

```javascript
// 使用 new_task 启动独立上下文
// ⚠️ 重要：子任务在全新上下文中执行，无父任务对话历史
new_task({
  mode: "code",
  message: `
## 任务：${taskId} - ${taskName}

⚠️ **这是一个独立的上下文环境**
- 你没有父任务的对话历史
- 请使用 agent-workflow 技能来执行此任务
- 所有必需信息都在本消息中提供

### 上下文
- 项目：${projectName}
- 阶段：${phase}
- 依赖：${dependencies}
- 状态文件位置：.roo/gsd-state/STATE.md

### 要求
${taskDescription}

### 验收标准
- [ ] ${criteria1}
- [ ] ${criteria2}

### 执行指引
1. 首先读取状态文件了解项目上下文
2. 按照 agent-workflow 技能执行任务
3. 完成后更新 STATE.md
4. Git 原子提交
5. 使用 attempt_completion 返回结果
  `,
  todos: `
  - [ ] 读取状态文件恢复上下文
  - [ ] 步骤 1
  - [ ] 步骤 2
  - [ ] 验证
  - [ ] 更新状态文件
  `
})
```

### 阶段五：验证（Verify）

**目的**: 确保目标达成，而非仅任务完成

**验证层级**:

```
┌─────────────────────────────────────────┐
│           目标验证                       │
│   用户价值是否实现？                      │
└─────────────────────────────────────────┘
              │
┌─────────────────────────────────────────┐
│           功能验证                       │
│   所有功能按预期工作？                    │
└─────────────────────────────────────────┘
              │
┌─────────────────────────────────────────┐
│           质量验证                       │
│   代码质量、测试覆盖、文档完整？          │
└─────────────────────────────────────────┘
```

**验证清单**:

```markdown
## 功能验证
- [ ] 所有 REQ 标准满足
- [ ] 边界情况处理
- [ ] 错误处理完善
- [ ] 无回归问题

## 质量验证
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 代码规范检查通过
- [ ] 无安全漏洞

## 文档验证
- [ ] README 更新
- [ ] API 文档更新
- [ ] CHANGELOG 更新
```

## 快速模式

**适用场景**: 小任务、单一功能、快速修复

**简化流程**:

```
/gsd:quick [--discuss] [--full]

--discuss  包含讨论阶段
--full     完整验证流程
```

**执行**:

```markdown
1. 快速需求确认（1-2 轮问答）
2. 直接进入执行
3. 简化验证（核心功能测试）
4. 提交完成
```

## 波浪式并行执行

### 概念

分析任务依赖关系，将独立任务分组，每组（波浪）并行执行。

### 实现示例

```javascript
// 分析依赖关系
const taskGraph = {
  'A1': [],
  'A2': [],
  'A3': ['A1'],
  'A4': ['A1', 'A2'],
  'A5': ['A3'],
  'A6': ['A4'],
  'A7': ['A5', 'A6']
}

// 波浪分组
const waves = [
  ['A1', 'A2'],           // 第一波：无依赖
  ['A3', 'A4'],           // 第二波：依赖第一波
  ['A5', 'A6'],           // 第三波：依赖第二波
  ['A7']                  // 第四波：依赖第三波
]

// 执行
for (const wave of waves) {
  // 并行启动当前波浪的所有任务
  await Promise.all(wave.map(task => executeTask(task)))
}
```

## 注意事项

1. **每个任务独立上下文** - 通过文件恢复状态，不依赖对话历史
2. **原子提交** - 每个任务单独 Git 提交，保持历史清晰
3. **验证是必须的** - 不跳过验证环节
4. **状态同步** - 实时更新 STATE.md，便于恢复
5. **灵活调整** - 计划可根据实际情况调整，但要记录原因
6. **失败记录** - 记录失败原因和重试策略
7. **文件冲突检测** - 并行任务前检查文件所有权

## 失败恢复机制

### 任务失败处理流程

```
任务执行失败
    │
    ▼
┌─────────────────┐
│ 记录失败信息     │
│ - 错误信息       │
│ - 失败时间       │
│ - 影响范围       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 分析失败类型     │
└────────┬────────┘
         │
    ┌────┴────┬──────────┬──────────┐
    ▼         ▼          ▼          ▼
 可重试    需修改计划   需人工介入   致命错误
    │         │          │          │
    ▼         ▼          ▼          ▼
 自动重试  调整PLAN.md  标记阻塞   汇报用户
```

### 失败类型分类

```markdown
| 类型 | 特征 | 处理方式 |
|------|------|----------|
| 临时失败 | 网络超时、资源临时不可用 | 自动重试（最多3次） |
| 依赖失败 | 上游任务未完成或失败 | 等待依赖修复 |
| 规划错误 | 任务定义不清晰、范围错误 | 修改 PLAN.md |
| 技术障碍 | 技术方案不可行 | 切换方案或请求帮助 |
| 致命错误 | 无法自动恢复 | 标记阻塞，汇报用户 |
```

### STATE.md 失败记录格式

```markdown
# 项目状态

## 当前阶段
Phase 1: 核心功能

## 进度
- 总任务: 20
- 完成: 15
- 失败: 2
- 阻塞: 1
- 待开始: 2

## 失败任务

### A3: API 开发
- **状态**: failed
- **失败时间**: 2026-03-13 14:30
- **错误类型**: 依赖失败
- **错误信息**: 数据库连接配置缺失
- **影响任务**: A5, A6
- **重试策略**: 
  1. 等待 A1 修复
  2. 添加数据库配置
  3. 重新执行 A3

### A7: 性能优化
- **状态**: retry
- **失败次数**: 1
- **错误类型**: 临时失败
- **重试计划**: 自动重试中

## 阻塞项
- A5: 集成测试 - 等待 A3 完成
- A6: 文档编写 - 等待 A3 完成
```

### 断点续传

```markdown
## 恢复执行流程

1. 读取 STATE.md 了解当前状态
2. 检查是否有失败/阻塞任务
3. 确定下一个可执行任务
4. 从失败点继续

## 恢复命令

/gsd:resume [phase-number]

自动操作：
- 读取 STATE.md
- 跳过已完成任务
- 从第一个未完成任务开始
- 恢复失败任务（如果是可重试类型）
```

## 并发安全策略

### 文件所有权声明

在 PLAN.md 中明确每个任务的文件范围：

```markdown
## 文件所有权

| 任务 | 拥有文件 | 只读文件 |
|------|----------|----------|
| A1 | src/db/*.sql | - |
| A2 | src/frontend/** | src/types/* |
| A3 | src/api/auth.ts | src/db/schema.sql |

## 冲突检测
- A2 和 A3 都需要 src/types/user.ts → 放入不同波浪
```

### 状态文件分区

```markdown
# STATE.md 分区结构

## 任务状态区（每个任务更新自己的区域）

### A1 状态

- 更新人: agent-a1
- 更新时间: 2026-03-13 10:00

### A2 状态
- 进度: 进行中
- 更新人: agent-a2
- 更新时间: 2026-03-13 11:00

## 全局状态区（协调器更新）
```
