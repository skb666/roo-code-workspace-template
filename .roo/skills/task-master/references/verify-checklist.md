## PDCA 持续改进循环

### 定义

PDCA（Plan-Do-Check-Act）是由质量管理专家戴明提出的持续改进循环模型，是项目管理的核心方法论之一。

### PDCA 四阶段

```
         ┌──────────────────────────────────┐
         │                                  │
         ▼                                  │
    ┌─────────┐                        ┌─────────┐
    │  Plan   │                        │   Act   │
    │  规划   │                        │  改进   │
    └────┬────┘                        └────┬────┘
         │                                  │
         ▼                                  │
    ┌─────────┐                        ┌─────────┐
    │   Do    │───────────────────────▶│  Check  │
    │  执行   │                        │  检查   │
    └─────────┘                        └─────────┘
```

### 各阶段详解

| 阶段 | 英文 | 核心问题 | 关键活动 |
|------|------|----------|----------|
| **P** | Plan | 目标是什么？怎么做？ | 设定目标、制定计划、分解任务 |
| **D** | Do | 执行了吗？过程如何？ | 按计划执行、记录数据、跟踪进度 |
| **C** | Check | 结果如何？有什么偏差？ | 对比目标、分析偏差、识别问题 |
| **A** | Act | 如何改进？经验是什么？ | 纠正偏差、总结经验、标准化 |

### PDCA 与 Task Master 四阶段对应

| PDCA | Task Master 阶段 | 主要工具 |
|------|------------------|----------|
| Plan | 阶段一：任务拆解 | WBS + MECE + SMART |
| Do | 阶段二：规划执行 | 诺伊曼思维 + 优先级 |
| Check | 阶段三：验证 | 检查清单 + 偏差分析 |
| Act | 阶段四：汇报 + 改进 | 金字塔原理 + 复盘 |

### PDCA 循环特点

1. **持续循环**：一次循环结束，开始下一次循环
2. **螺旋上升**：每次循环都带来改进和提升
3. **闭环管理**：问题有始有终，形成闭环
4. **全员参与**：需要所有相关人员共同参与

### PDCA 实施示例

```
场景：登录功能开发延期

Plan（规划）：
- 目标：3 月 28 日上线登录功能
- 计划：前端 2 周 + 后端 2 周 + 测试 1 周

Do（执行）：
- 前端 3/15 开始，后端 3/15 开始
- 实际执行中发现第三方登录接入困难

Check（检查）：
- 3/22 检查：后端 API 完成，前端延期 2 天
- 偏差原因：第三方登录文档不全，接入时间超出预期
- 影响：整体上线可能延期 2 天

Act（改进）：
- 短期措施：协调技术专家支援，加班赶回进度
- 长期改进：建立第三方服务接入评估流程
- 标准化：更新《第三方服务接入指南》

下一轮 PDCA：
- Plan：基于本次经验，重新评估其他第三方服务接入时间
```

## 验证的目的与层次

### 验证的目的

验证是确保项目按计划进行、交付成果符合预期的关键环节。

**验证的核心问题**：
- 我们是否在做正确的事？（效果）
- 我们是否正确地做事？（效率）
- 我们是否完成了所有应该完成的事？（完整性）

### 验证的层次

#### 层次 1：工作包级验证

每个工作包完成后立即验证。

**验证内容**：
- [ ] 交付成果是否符合 SMART 目标的衡量标准？
- [ ] 质量是否达到验收标准？
- [ ] 时间和成本是否在估算范围内？
- [ ] 是否有遗留问题？

**验证人**：工作包负责人 + 直接主管

**验证时机**：工作包完成后、下一工作包开始前

---

#### 层次 2：任务级验证

一个完整任务（包含多个工作包）完成后验证。

**验证内容**：
- [ ] 该任务下所有工作包是否都已完成？
- [ ] 工作包之间的接口/依赖是否正确？
- [ ] 整体交付成果是否符合预期？
- [ ] 是否有跨工作包的问题？

**验证人**：任务负责人 + 项目经理

**验证时机**：任务所有工作包完成后

---

#### 层次 3：项目级验证

项目所有任务完成后、正式上线前验证。

**验证内容**：
- [ ] WBS 中所有任务是否都已完成？
- [ ] 所有交付成果是否齐全？
- [ ] 是否通过所有测试？
- [ ] 文档是否完整？
- [ ] 是否具备上线条件？

**验证人**：项目经理 + 关键干系人

**验证时机**：项目交付前

---

#### 层次 4：运营级验证

上线后持续监控和验证。

**验证内容**：
- [ ] 系统运行是否正常？
- [ ] 用户反馈如何？
- [ ] 关键指标是否达标？
- [ ] 是否有新的问题？

**验证人**：运维团队 + 产品经理

**验证时机**：上线后持续

## 检查清单设计

### 设计原则

1. **具体明确**：每项检查内容清晰不含糊
2. **可操作**：检查项可直接执行，无需额外解释
3. **完整覆盖**：覆盖所有关键风险点
4. **适度粒度**：不过粗（遗漏风险）也不过细（难以执行）
5. **持续更新**：根据实际情况不断优化

## 偏差分析与根因分析

### 偏差识别

| 偏差类型 | 识别信号 | 严重程度判断 |
|----------|----------|--------------|
| 进度偏差 | 实际完成日期晚于计划日期 | 延期 < 10% 低，10-30% 中，> 30% 高 |
| 成本偏差 | 实际花费超出预算 | 超支 < 10% 低，10-30% 中，> 30% 高 |
| 质量偏差 | 测试不通过、缺陷率高 | 缺陷数 < 5 低，5-20 中，> 20 高 |
| 范围偏差 | 交付成果与计划不一致 | 变更 < 10% 低，10-30% 中，> 30% 高 |
| 资源偏差 | 实际使用资源与计划不符 | 偏差 < 20% 低，20-50% 中，> 50% 高 |

### 5 Whys 根因分析法

**定义**：通过连续问"为什么"（通常 5 次）找出问题的根本原因，而非停留在表面原因。

**核心原则**：
- 不要满足于表面答案
- 追问直到找到可操作的根因
- 根因应该是可以改变的因素

#### 5 Whys 示例

```
问题：登录功能开发延期 2 天

为什么 1？→ 因为 API 开发比预计多花了 2 天
    │
为什么 2？→ 因为遇到了技术难题，需要研究新方案
    │
为什么 3？→ 因为需求中使用了团队不熟悉的第三方登录技术
    │
为什么 4？→ 因为技术选型时未充分评估团队能力
    │
为什么 5？→ 因为技术评审流程不完善，缺乏能力评估环节

═════════════════════════════════════════════════════
根本原因：技术评审流程不完善，缺乏能力评估环节
纠正措施：在技术评审流程中增加团队能力评估环节
预防措施：建立新技术引入评估机制
═════════════════════════════════════════════════════
```

### 鱼骨图分析（因果分析）

用于多因素复杂问题的根因分析：

```
                    问题：登录功能延期
                           │
    ┌──────────────────────┼──────────────────────┐
    │                      │                      │
  人员                    方法                    机器
    │                      │                      │
    ├─ 技能不足            ├─ 流程不规范          ├─ 开发环境不稳定
    └─ 人员变动            └─ 估算不准确          └─ 工具不熟练
    
    ┌──────────────────────┼──────────────────────┐
    │                      │                      │
  材料                    环境                    测量
    │                      │                      │
    ├─ 文档不全            ├─ 需求频繁变更        ├─ 验收标准不明确
    └─ 第三方依赖问题      └─ 沟通不畅            └─ 进度跟踪不及时
```

### 纠正措施选择

| 偏差类型 | 纠正措施选项 | 适用条件 |
|----------|--------------|----------|
| 进度滞后 | 增加资源 | 有额外人力可用 |
|          | 加班赶工 | 短期偏差，团队可承受 |
|          | 缩减范围 | 非核心功能可删减 |
|          | 调整依赖 | 有并行空间 |
| 成本超支 | 削减非核心功能 | 预算固定 |
|          | 寻找替代方案 | 有性价比更高的方案 |
|          | 申请追加预算 | 有合理理由 |
| 质量问题 | 返工修复 | 问题可修复 |
|          | 增加测试 | 需要更多验证 |
|          | 引入专家评审 | 内部能力不足 |
| 范围变更 | 评估影响 | 变更前必做 |
|          | 更新计划 | 变更确认后 |
|          | 获得干系人确认 | 重要变更 |

## 复盘与经验总结

### 复盘的定义

复盘是对已完成工作的系统回顾，目的是总结经验教训，改进未来工作。

### 复盘四步法

```
┌─────────────────────────────────────────────────────┐
│                    复盘四步法                        │
├─────────────────────────────────────────────────────┤
│ 1. 回顾目标：原定目标是什么？                        │
│ 2. 评估结果：实际结果如何？与目标差异？              │
│ 3. 分析原因：为什么会有差异？成功/失败的关键因素？   │
│ 4. 总结规律：有什么经验教训？下次如何改进？          │
└─────────────────────────────────────────────────────┘
```

## 验证会议

### 高效验证会议要点

**会前准备**：
- [ ] 负责人提前提交交付成果
- [ ] 验证人员提前审阅材料
- [ ] 准备检查清单
- [ ] 预约会议室/视频会议

**会议进行**：
- [ ] 按检查清单逐项核对
- [ ] 记录问题和偏差
- [ ] 明确纠正措施和责任人
- [ ] 控制会议时间（建议 < 1 小时）

**会后跟进**：
- [ ] 发送会议纪要
- [ ] 跟踪纠正措施执行
- [ ] 更新项目状态

关键路径：后端 API → 前端联调 → 集成测试 → 上线

═══════════════════════════════════════════════════════
Do（执行）
═══════════════════════════════════════════════════════
执行过程：
- 3/15：项目启动，开始开发
- 3/17：后端 API 开发遇到第三方登录问题
- 3/18：协调专家支援，问题解决
- 3/20：后端 API 完成

- 3/23：开始集成测试

- 前端按计划完成
- 测试发现 3 个 Bug（1 个高优，2 个低优）

═══════════════════════════════════════════════════════

═══════════════════════════════════════════════════════
结果对比：

|------|------|------|------|

═══════════════════════════════════════════════════════
